# 14 数据访问层设计

## 1. 文档定位

本文档定义数据访问层的轻量设计规范。

目标不是描述业务功能，而是约束：

- Repository 如何拆分
- 查询和写入接口如何设计
- 哪些逻辑可以放在数据访问层
- 哪些逻辑不能放在数据访问层

## 2. 层次定位

数据访问层负责：

- Repository
- SQL
- 批量写入
- 范围查询
- 索引命中
- 事务边界内的数据持久化

数据访问层不负责：

- 页面逻辑
- 同步编排
- 新鲜度判断
- 查询降级策略
- 业务 DTO 拼装

## 3. Repository 拆分原则

### 3.1 按责任拆分

优先按数据责任拆分，而不是按页面拆分。

推荐拆分：

- `StockBarsRepository`
- `FuturesBarsRepository`
- `MarketSyncJobRepository`
- `MarketQualityRepository`
- `MonitorRepository`
- `BluechipPoolRepository`
- `BluechipResultRepository`
- `StockBasicsRepository`
- `FuturesBasicsRepository`

### 3.2 不按页面拆分

不建议出现：

- `MarketDataPageRepository`
- `StockMonitorPageRepository`
- `BluechipPageRepository`

页面只是调用方，不应该成为 Repository 边界。

## 4. 查询接口规范

### 4.1 查询接口应返回原始数据模型

Repository 返回：

- 行记录
- 聚合前基础数据
- 分页结果
- 轻量统计结果

Repository 不返回：

- 应用层 DTO
- warning/error 解释文案
- 业务展示结构

### 4.2 查询接口命名建议

常见命名：

- `findById`
- `findByCode`
- `findByCodes`
- `listByCategory`
- `listByRange`
- `listActive`
- `countByRange`
- `upsertMany`

### 4.3 查询条件保持稳定

查询条件建议以结构化参数表达，例如：

- `symbol`
- `timeframe`
- `startDate`
- `endDate`
- `limit`
- `offset`

不要把上层页面筛选参数原样透传成 SQL 拼接碎片。

## 5. 写入接口规范

### 5.1 批量写入优先

行情数据写入优先提供：

- `upsertMany`
- `insertMany`
- `replaceMany`

而不是高频单条循环写入。

### 5.2 幂等性要求

对于基础事实表和治理表：

- 必须支持幂等写入
- 依赖唯一键或主键约束去重
- 重复执行任务时不能造成脏数据堆积

### 5.3 更新时间统一处理

Repository 负责：

- `created_at`
- `updated_at`
- `synced_at` 等落库时间字段

上层不应频繁自己拼接这些字段。

## 6. 事务与一致性

### 6.1 使用事务的场景

建议在以下场景使用事务：

- 同步任务主表 + 子任务表同时写入
- 标的池主表 + 成分股表同时更新
- 批量补数结果写入 + 任务状态更新

### 6.2 不在 Repository 内做跨层补偿逻辑

Repository 可以做事务，但不负责：

- 重试策略
- 备源切换
- 新鲜度决策
- 业务失败兜底

这些属于上层服务职责。

## 7. 索引与查询约束

### 7.1 行情表约束

查询必须围绕既定索引路径组织：

- 股票：`stock_code + timeframe + trade_day + bucket_ts`
- 期货：`quote_code + timeframe + trade_day + bucket_ts`

### 7.2 列表表约束

配置表优先按以下字段组合读取：

- `category_id + sort_order + id`
- `is_active / is_enabled`

### 7.3 治理表约束

任务表和质量表优先按：

- `status`
- `job_type + created_at`
- `dataset_name + timeframe + generated_at`

设计索引和查询。

## 8. 禁止事项

数据访问层禁止：

- 拼业务说明文案
- 拼页面展示对象
- 决定是否切换第三方源
- 决定是否补数
- 决定是否返回 stale 数据
- 把不同页面逻辑混进 SQL 分支里

## 9. 与其它层的接口关系

对上：

- 数据管理层调用 Repository 完成落库、任务记录、质量报告写入
- 数据查询服务层调用 Repository 完成基础表读取和轻量统计查询

对下：

- 只面向数据库

## 10. 文档结论

数据访问层文档保持轻量即可，重点是把边界和规范说清：

- 哪些表由哪些 Repository 负责
- 查询和写入接口怎么设计
- 哪些逻辑不能下沉到 Repository

只要这层边界稳定，上面的数据管理层和数据查询服务层就不容易重新耦合。
