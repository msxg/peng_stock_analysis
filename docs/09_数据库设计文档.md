# 09 数据库设计文档

## 1. 文档说明

本文档描述 `peng_stock_analysis` 项目当前版本的数据库设计，基于 2026-05-13 代码状态整理，主要对应以下实现：

- 主库初始化：[src/db/database.js](/Users/ddxx/Dev/TestWs/peng_stock_analysis/src/db/database.js)
- 股票日线本地缓存表：[src/services/stockDataService.js](/Users/ddxx/Dev/TestWs/peng_stock_analysis/src/services/stockDataService.js)

当前系统使用 `SQLite + better-sqlite3`，以单文件数据库为主，默认开启：

- `PRAGMA journal_mode = WAL`
- `PRAGMA foreign_keys = ON`

设计目标：

- 支持单机部署和轻量维护
- 以“功能域拆分表”而不是“超大宽表”组织数据
- 对高频查询补充必要索引
- 对部分历史能力采用“追加写入 + 结果缓存”方式，降低实时分析成本

## 2. 总体分层

数据库按业务可分为 10 个域：

1. 认证与系统配置
2. 通用股票分析与回测
3. 持仓管理
4. 期货监测
5. 股票基础资料与行情监测
6. 蓝筹模式与标的池
7. AI 对话会话
8. 用量统计
9. 焦点资讯
10. 本地股票日线缓存

其中：

- `createSchema()` 负责主表初始化
- `ensureMigrations()` 负责增量补字段
- `stock_daily_bars` 不在主 schema 中，而是在股票行情服务首次使用本地日线缓存时惰性创建

## 3. 设计原则

### 3.1 主键策略

- 大多数业务表使用 `INTEGER PRIMARY KEY AUTOINCREMENT`
- 少数天然唯一实体使用业务键，例如：
  - `futures_basics.quote_code`
  - `news_scheduler_states.provider_key`

### 3.2 时间字段

- 大部分表统一包含 `created_at`
- 可变实体通常包含 `updated_at`
- 时间默认使用 `datetime('now')`
- 行情和业务日期类字段使用文本格式存储，例如 `YYYY-MM-DD`

### 3.3 关系约束

- 重要父子关系显式声明外键，并采用 `ON DELETE CASCADE`
- 对纯查询型、快照型、缓存型表，优先保证读写效率，不强依赖复杂外键网络

### 3.4 索引策略

- 列表管理表：按 `sort_order + id`
- 历史结果表：按 `stock_code + 日期倒序`
- 资讯表：按 `published_at`、分类、指纹等多维索引
- 蓝筹结果表：按 `batch_id`、分析日期、信号日期、股票代码建立索引

## 6. 初始化与种子数据

系统启动时会自动注入部分默认数据：

- 默认管理员用户 `admin`
- `auth_settings` 单行记录
- 系统配置默认项 `DEFAULT_SYSTEM_CONFIGS`
- 默认期货分类与示例合约
- 默认股票监测分类与示例股票
- 默认蓝筹标的池与示例成分
- 默认资讯源 `tushare`
- 默认资讯 taxonomy 树

设计意义：

- 新环境可以开箱即用
- 前端页面首次进入时就有可见数据结构
- 降低开发和测试环境初始化成本

## 7. 迁移策略

当前项目没有独立的迁移版本表，采用“启动时自检补字段”的轻量策略：

- 通过 `hasTable()` / `hasColumn()` 判断
- 缺失则执行 `ALTER TABLE`

当前已覆盖的迁移包括：

- `backtest_results.tp_hit`
- `backtest_results.sl_hit`
- `stock_basics` 多个扩展字段
- `futures_categories.is_enabled`
- `stock_monitor_categories.is_enabled`
- `news_items.meta_json`
- `news_provider_categories.scheduler_enabled`
- `news_provider_categories.scheduler_priority`
- `news_scheduler_states` 的调度状态字段

风险与建议：

- 这种方式适合当前单机、小团队阶段
- 当 schema 变更频率继续上升后，建议增加显式迁移版本表，例如 `schema_migrations`
- 对数据回填逻辑复杂的迁移，建议拆分成独立脚本，不只依赖启动时补字段

## 8. 当前设计的优点与边界

优点：

- 结构清晰，按业务域划分明显
- SQLite 足够支撑当前单机分析场景
- 蓝筹模式已具备标的池、结果保存、本地日线缓存三层数据基础
- 资讯模块的数据链路完整，适合继续做聚类、摘要、调度

边界：

- `analysis_history`、`bluechip_analysis_signals`、`stock_daily_bars` 都带有历史累积特征，后续需要归档或清理策略
- 一部分表之间是“逻辑关系”而非强外键，便于快照保存，但需要应用层保证一致性
- `stock_daily_bars` 目前只覆盖本地日线缓存，不是统一行情中心
- 缺少正式迁移版本控制，后续多人协作时维护成本会上升

1. 增加数据库版本表，规范迁移管理
2. 为 `bluechip_analysis_signals` 增加批次主表，例如 `bluechip_analysis_batches`，把统计信息和参数快照上移
3. 为历史大表增加清理策略，例如按月份归档或保留最近 N 个月
4. 将 `stock_daily_bars` 提升为统一市场数据子域，逐步覆盖更多市场和周期
5. 补充 ER 图和字段字典导出脚本，减少手工维护文档成本
