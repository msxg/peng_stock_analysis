## 2. 设计原则

### 2.1 分域设计

数据库按业务域和平台域拆分，不采用超大统一宽表。

### 2.2 市场数据分层

行情数据按“市场 + 层级”拆分：

- 股票与期货分表
- `intraday` 与 `eod` 分表
- 派生粒度默认实时聚合，按需物化

### 2.3 管理与查询分离

- 数据管理层需要治理表记录同步、补数、质量巡检
- 数据查询服务层消费基础事实表，不直接承担治理记录写入

### 2.4 配置与结果分离

- 监控配置、标的池配置、分析结果分开建模
- 快照结果优先保留独立结果表，避免污染配置主表

### 2.5 索引优先支持高频路径

索引主要围绕：

- 标的/周期/日期范围查询
- 列表管理排序
- 批次和历史结果回放
- 同步任务状态追踪

## 3. 数据域划分

数据库按 10 个子域组织：

1. 认证与系统配置
2. 通用分析与回测
3. 持仓管理
4. 市场基础资料
5. 统一监控配置
6. 市场行情事实表
7. 市场数据治理
8. 蓝筹模式与标的池
9. AI 对话与用量
10. 焦点资讯

## 4. 核心市场数据子域

### 4.1 市场基础资料

股票：

- `stock_basics`

期货：

- `futures_basics`

用途：

- 保存名称、市场、交易所、交易时段、基础属性
- 为搜索、展示、同步和查询路由提供基础信息

### 4.2 统一监控配置

- `monitor_categories`
- `monitor_symbols`

用途：

- 管理监控分类
- 管理股票/期货/指数等统一监控标的
- 为 `行情监测` 提供配置入口

### 4.3 市场行情事实表

股票：

- `stock_intraday_bars`
- `stock_eod_bars`

期货：

- `futures_intraday_bars`
- `futures_eod_bars`

原则：

- `1m` 和 `1d` 作为基础粒度
- `5m/15m/30m/60m` 默认从 `1m` 聚合
- `1w/1M/1Y` 默认从 `1d` 聚合
- 性能不足时可在原表物化更高粒度

### 4.4 市场数据治理

- `market_sync_jobs`
- `market_sync_job_items`
- `market_data_quality_reports`

用途：

- 记录同步任务、补数任务、查询触发刷新任务
- 记录子任务执行结果和失败原因
- 记录覆盖率、缺口和异常值巡检结果

### 4.5 蓝筹模式与结果沉淀

- `bluechip_pools`
- `bluechip_pool_symbols`
- `bluechip_analysis_signals`

用途：

- 管理标的池配置
- 管理成分股
- 沉淀信号结果快照

## 6. 关键表设计

### 6.1 `monitor_categories`

用途：

- 统一监控分类主表

关键字段：

- `name`
- `description`
- `sort_order`
- `is_enabled`
- `created_at`、`updated_at`

索引：

- 分类名称唯一约束

### 6.2 `monitor_symbols`

用途：

- 统一监控标的配置表

关键字段：

- `category_id`
- `symbol_code`
- `quote_code`
- `symbol_type`
- `market`
- `exchange`
- `display_name`
- `sort_order`
- `is_active`

关系：

- `category_id -> monitor_categories.id ON DELETE CASCADE`

索引：

- `idx_monitor_symbols_category`
- `idx_monitor_symbols_active`
- 唯一约束：`(category_id, symbol_type, market, symbol_code)`

### 6.3 `stock_intraday_bars`

唯一键：

索引：

## 7. 关键关系摘要

核心关系：

- `monitor_categories 1 -> N monitor_symbols`
- `market_sync_jobs 1 -> N market_sync_job_items`
- `bluechip_pools 1 -> N bluechip_pool_symbols`
- `bluechip_pools 1 -> N bluechip_analysis_signals`：以 `pool_id` 和快照字段形成逻辑关联
- `analysis_history 1 -> N backtest_results`
- `chat_sessions 1 -> N chat_messages`

## 8. 索引策略

### 8.1 行情事实表

核心索引围绕：

- `symbol/quote_code + timeframe + trade_day + bucket_ts`
- 支持范围查询、聚合前基础读取、倒序回放

### 8.2 列表与配置表

核心索引围绕：

- `sort_order + id`
- `is_active / is_enabled`
- 分类下标的顺序读取

### 8.3 治理表

核心索引围绕：

- 任务状态
- 任务类型与创建时间
- 子任务状态
- 巡检报告的 `dataset + timeframe + generated_at`

### 8.4 结果快照表

核心索引围绕：

- 分析日期
- 信号日期
- 批次 ID
- 股票代码
- 标的池编码

## 9. 演进建议

1. 治理表优先按任务和巡检结果建模，不把治理信息混入基础事实表
2. 统一监控配置优先使用 `monitor_categories/monitor_symbols`，不再区分股票监控表和期货监控表
3. `intraday/eod` 四张基础事实表作为长期稳定的市场数据底座
4. 若后续查询压力增大，再评估物化更高粒度，不额外引入大量派生表
5. 蓝筹模式后续可增加批次主表，例如 `bluechip_analysis_batches`
