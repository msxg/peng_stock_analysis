# 焦点资讯技术方案与开发计划

## 1. 背景与目标

“焦点资讯”模块不是简单的新闻列表，而是面向后续 AI 提炼、监控联动、分类观察、自选池关联的一套资讯数据平台。其核心目标不是“临时抓几条新闻”，而是：

- 构建项目自有的资讯数据资产库
- 建立项目自有的资讯分类体系，并支持与外部来源分类映射
- 抽象统一的 `data provider` 层，使 `tushare` 成为首个实现，而不是唯一实现
- 以定时采集为主，沉淀历史资讯资产
- 为后续实时查询、事件聚类、AI 摘要与推送预留架构空间

## 2. 设计原则

- 资产化优先：所有资讯先入库，再提供查询与 AI 使用
- Provider 解耦：任何外部来源都必须先适配到统一模型
- 分类自治：项目有自己的分类树，不直接绑定单一 provider 的分类结构
- ETL 分层：抽取、清洗、标准化、入库、聚类、摘要职责分离
- 先离线后实时：一期以定时采集和库内查询为主，实时查询只做扩展位
- 幂等与可回溯：原始响应、标准化记录、同步任务状态都必须可追踪
- fail-open：单一 provider 或单次同步失败不阻断整个系统

## 3. 范围与非目标

### 一期范围

- 建立资讯分类资产
- 建立 provider 适配层
- 接入 `tushare` 作为首个 provider
- 支持定时采集、手动采集、增量同步
- 建立原始表、标准表、分类映射表、同步记录表
- 提供资讯库查询页面与基础 API
- 建立基础去重、实体关联、焦点评分能力
- 建立 AI 摘要任务模型与首个“每日焦点摘要”能力

### 一期非目标

- 不以网页实时抓取 `https://tushare.pro/news/` 作为主要实现
- 不优先做跨 provider 的全文语义搜索
- 不优先做 WebSocket 级实时推送
- 不优先做复杂的多轮问答式资讯 Agent

## 4. 总体架构

整体按五层设计：

1. `Provider Layer`
   - 定义标准 provider 接口
   - `tushare` 为首个实现
   - 后续可增加交易所、公告站点、第三方财经媒体等

2. `ETL Layer`
   - 负责抽取、标准化、映射、去重、入库
   - 负责同步任务调度、失败重试、窗口增量

3. `News Asset Layer`
   - 存放原始资讯、标准化资讯、分类、映射、实体关联、事件簇、AI 摘要
   - 作为页面查询与 AI 分析的唯一可信数据源

4. `Query Layer`
   - 提供资讯列表、分类浏览、关键词检索、按自选池关联查询
   - 默认只查资产库，不直接透传 provider

5. `AI Layer`
   - 基于标准化资讯和事件簇生成焦点摘要
   - 输出“今日焦点”“分类焦点”“自选池焦点”“风险提醒”

## 5. 与现有项目架构的衔接

建议沿用当前项目的分层模式：

## 6. 分类体系设计

### 6.1 总体思路

- `tushare` 的“资讯数据”页面分类体系作为一期基准参考
- 项目内部维护自己的标准分类树
- provider 原始分类与项目标准分类通过映射关系关联

### 6.2 分类模型

建议拆成三套概念：

- `provider taxonomy`
  - 例如 `tushare.news`、`tushare.major_news`、`tushare.cctv_news`
  - 表示 provider 原生接口域和分类域

- `internal taxonomy`
  - 项目自有分类树
  - 例如：
    - 宏观政策
    - 市场快讯
    - 上市公司公告
    - 交易所互动
    - 行业资讯
    - 海外市场

- `taxonomy mapping`
  - 负责从 provider 分类映射到内部分类
  - 支持一对一、一对多、多对一

### 6.3 分类层级建议

- 一级分类：资讯大类
- 二级分类：来源域/主题域
- 三级标签：场景标签，可选

示例：

- 一级：宏观政策
- 二级：新闻联播 / 部委动态 / 海外宏观
- 标签：货币政策 / 地产 / 出海 / AI / 锂电

## 9. ETL 设计

### 9.1 同步模式

支持三种模式：

- `bootstrap`
  - 首次全量同步分类与最近窗口资讯

- `incremental`
  - 定时增量同步
  - 一期主模式

- `backfill`
  - 历史回补
  - 手动触发

### 9.2 ETL 流程

1. 拉取 provider 分类目录
2. 更新 `news_provider_categories`
3. 按分类拉取资讯数据
4. 原始响应写入 `news_raw_items`
5. 标准化生成 `news_items`
6. 做精确去重
7. 生成事件指纹并尝试聚类
8. 建立分类映射与实体关联
9. 更新统计并写回 `news_sync_runs`
10. 如满足条件，触发 AI 摘要任务

### 9.3 幂等策略

- provider 主键幂等：
  - `provider_key + provider_item_id`
- 内容指纹幂等：
  - `title + source + published_at` 归一化后计算 `dedupe_fingerprint`
- 事件聚类指纹：
  - `主题词 + 关键实体 + 时间窗口` 计算 `event_fingerprint`

### 9.4 去重策略

分三层：

- 原始去重
  - 同 provider 同 item id
- 标题去重
  - 标题归一化后完全一致
- 事件聚合
  - 标题相似、实体重合、时间接近则归入同事件簇

### 9.5 重要性评分

建议一期用规则模型：

- 分类权重
- 来源权重
- 时效权重
- 是否命中监控品种/分类
- 是否为公告/互动问答/政策类
- 是否多源重复出现

后续可升级为“规则分 + AI 调整分”混合模型。

## 10. 实体关联设计

资讯需要和现有项目资产建立关系：

- 股票
- 指数
- 期货
- 市场
- 行业/概念
- 行情监测分类

实体识别分两步：

1. 规则抽取
   - 基于标题、正文、标签、公告代码
2. 资产映射
   - 对接现有 `stock_basics`
   - 对接现有监控分类与品种配置

这样后续即可实现：

- “某分类的焦点资讯”
- “自选池相关资讯”
- “公告触发的行情监测联动”

## 11. 调度与运行设计

### 11.1 一期选型

基于当前项目单进程 + SQLite 架构，一期建议：

- 在 API 进程内实现 `newsSchedulerService`
- 使用分钟级 tick + 配置规则判断是否触发同步
- 通过数据库锁或运行中状态防止重复执行

建议不要一开始引入独立 worker 集群。

### 11.2 调度类型

- 启动补齐
  - 系统启动时检查分类与最近窗口数据

- 周期同步
  - 如每 15 分钟同步快讯
  - 每天固定时段同步长文/公告/互动问答

- 手动同步
  - 页面点击触发

- 历史回补
  - 指定 provider、分类、时间窗口触发

### 11.3 配置项建议

建议新增以下系统配置：

- `NEWS_SYNC_ENABLED`
- `NEWS_SYNC_TICK_SECONDS`
- `NEWS_SYNC_LOOKBACK_MINUTES`
- `NEWS_BOOTSTRAP_DAYS`
- `NEWS_AI_DIGEST_ENABLED`
- `NEWS_AI_DIGEST_SCHEDULE`

## 13. AI 分析设计

### 13.2 输入对象

- `news_clusters`

- 经过去重的 `news_items`

### 13.3 输出类型

建议支持四类 digest：

- `daily_market_focus`
- `taxonomy_focus`
- `watchlist_focus`
- `risk_alert`

{

## 16. 里程碑验收标准

### M1：分类底座可用

- 能同步 `tushare` 分类目录
- 能查看 provider 分类与内部分类映射

### M2：资讯资产可用

- 能手动同步资讯
- 同一批数据重复同步不产生脏重复
- 能按时间、分类、来源查询

### M3：页面可用

- `/focus-news` 可访问
- 支持基础筛选
- 支持查看详情

### M4：自动化可用

- 定时任务可稳定运行
- 失败 run 可追踪
- 有同步统计和错误信息

### M5：AI 输出可用

- 能生成每日焦点摘要
- 能生成分类焦点
- 摘要可回溯到原始资讯与事件簇

## 17. 风险与应对

### 17.1 Tushare 权限边界

风险：

- 并非所有资讯域都一定具备权限

应对：

- provider 初始化阶段先做接口可用性探测
- 分类资产与 provider 实现分离，不因单域不可用阻断整体设计

### 17.2 无统一原文 URL

风险：

- 部分接口可能没有可直接跳转的原文链接

应对：

- 允许 `url` 为空
- 详情页优先展示标准化正文与来源信息

### 17.3 数据噪音高

风险：

- 快讯和互动问答噪音较大

应对：

- 一期先做规则过滤
- 二期引入聚类与 AI 重要性判断

### 17.4 单进程调度限制

风险：

- 单进程内调度在多实例部署时会重复执行

应对：

- 一期通过数据库锁控制
- 后续如多实例部署，再拆分独立 worker

## 18. 推荐实施顺序

建议按以下顺序推进：

1. 先做分类资产和 provider 抽象
2. 再做 `tushare` provider 与 ETL 入库
3. 再做资讯库页面与查询 API
4. 再做调度器
5. 最后做 AI 摘要和事件聚类

这样可以确保每一阶段都能形成独立可验收成果，且不会因为 AI 能力或实时能力尚未完善而阻塞主线。
